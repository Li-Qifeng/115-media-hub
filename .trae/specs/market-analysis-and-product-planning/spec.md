# 市场调研与产品规划 Spec

## Why

`115 Media Hub` 当前已是一个可运行的多网盘媒体自动化面板（v0.4.11，支持 115/Quark/天翼/123/阿里五家网盘、STRM 生成、TG 资源同步、影视订阅、刮削管理），但缺少基于行业实证的横向参照。为了避免"凭经验臆测"地堆功能，需要在动手扩展之前，先系统性调研 GitHub 上同领域（网盘媒体管理 / STRM 生成 / 资源订阅 / 影视刮削 / 自动化追更）的真实开源项目，提炼事实标准与反模式，再据此规划一个更优的演进方案。

本 spec 的产出是一组**研究/规划文档**（不是代码），用于指导后续 v0.5 / v1.0 / v2.0 的架构与功能决策。

## What Changes

- 新增 `docs/market-analysis.md`：≥20 个 GitHub 同类项目调研，按统一格式逐项分析定位/功能/架构/优缺点/社区。
- 新增 `docs/competitor-comparison.md`：横向对比矩阵 + 共性/独有创新/被频繁要求但没做好/被吐槽/被放弃的设计清单。
- 新增 `docs/best-practice.md`：事实标准 / 优秀实践 / 应避免设计 / 未来更值得采用方案，附原因与依据。
- 新增 `docs/product-plan.md`：基于调研设计一个更优方案的功能规划（P0/P1/P2/P3），含优先级理由/依赖/复杂度/风险。
- 新增 `docs/architecture.md`：新方案总体架构、模块设计、目录结构、插件/配置/权限/数据模型/API/事件/日志/测试/文档体系。
- 新增 `docs/roadmap.md`：v0.1 / v0.5 / v1.0 / v2.0 四个版本规划，每个版本含新增功能/重构/兼容策略/风险/交付目标。

**BREAKING**：无（纯文档产出，不改动现有代码）。

## Impact

- Affected specs：无（首批规划文档）。
- Affected code：不直接改动代码；产出供后续开发参考。新增文件全部位于 `docs/` 目录下，与现有 `docs/DEVELOPMENT.md`、`docs/superpowers/` 并列。
- 数据来源约束：所有分析必须基于 GitHub 实际项目（仓库链接、Star、最近 commit 时间、Issue/PR 数等可核验信息），引用分析依据，不得凭空臆测。

## ADDED Requirements

### Requirement: 同类项目调研覆盖度

系统 SHALL 调研不少于 20 个 GitHub 同领域开源项目，覆盖：
- 网盘媒体管理 / STRM 生成类
- 影视资源订阅 / 自动追更类
- 影视刮削 / 元数据管理类
- 网盘资源搜索 / 转存类
- 媒体服务器（Emby/Jellyfin/Plex）生态工具类
- 不同语言实现（Python / Go / Node / .NET 等）
- 不同架构（单体 / 插件化 / 微服务）

#### Scenario: 项目筛选达标
- **WHEN** 调研完成
- **THEN** 每个项目均包含可核验信息：仓库名、Star 数（近似）、最近一年活跃情况、是否文档完善
- **AND** 总数 ≥ 20

#### Scenario: 覆盖多样性
- **WHEN** 审视项目列表
- **THEN** 至少出现 3 种不同编程语言实现
- **AND** 至少出现 2 种不同架构形态（如单体 vs 插件化）

### Requirement: 项目分析统一格式

每个被调研项目 SHALL 按统一字段分析：项目定位、解决的问题、核心功能、技术架构、模块划分、插件机制、配置方式、API 设计、UI 设计（如有）、数据模型、扩展能力、优点、缺点、社区评价、常见 Issue、未来发展方向。

### Requirement: 横向对比矩阵

`competitor-comparison.md` SHALL 包含一张对比矩阵，至少覆盖：功能支持与否、实现方式、优缺点、复杂度、维护成本。并 SHALL 明确列出：大多数项目都有的功能、少数项目独有的创新、被用户频繁要求但没人做好、被大量吐槽的问题、被放弃的设计。

### Requirement: 最佳实践提炼有据

`best-practice.md` 的每条结论 SHALL 标注"成为事实标准 / 优秀实践 / 应避免 / 未来更值得采用"分类，并说明原因与引用依据（来自哪些项目的实证）。

### Requirement: 新方案设计完整

`architecture.md` SHALL 覆盖：总体架构、模块设计、目录结构、插件体系、配置体系、权限体系、数据模型、API 规范、事件系统、日志系统、测试方案、文档结构。新方案 SHALL 结合所有优秀项目优点而非复制单一项目。

### Requirement: 功能优先级与路线图

`product-plan.md` SHALL 按 P0/P1/P2/P3 输出功能优先级，每项说明为什么放在该优先级、依赖哪些模块、预计开发复杂度、风险。`roadmap.md` SHALL 覆盖 v0.1 / v0.5 / v1.0 / v2.0 四个版本，每个版本含新增功能、重构内容、兼容策略、风险分析、交付目标。

### Requirement: 引用可核验

所有文档 SHALL 在关键论断处引用具体仓库（GitHub URL / 项目名）作为分析依据，不得泛泛而谈。

## MODIFIED Requirements

无（首批规划文档，无既有需求被修改）。

## REMOVED Requirements
