# Roadmap：115 Media Hub 版本演进规划

> 基于 P0-P3 功能优先级规划 v0.1 / v0.5 / v1.0 / v2.0 四个版本演进。
> 调研时间：2026-07
> 原则：每个版本可独立交付、向后兼容、风险可控。

---

## 1. 版本规划原则

### 1.1 版本号语义

本路线图采用语义化版本思想，结合 NAS 用户"升级意愿低、升级失败丢数据是口碑杀手"的现实约束（应避免 3.8），定义版本号语义如下：

| 版本段 | 语义 | 兼容承诺 | 适用阶段 |
|--------|------|---------|---------|
| **v0.x** | 早期快速迭代 | 可有不兼容变更，但**必须**提供迁移脚本与迁移说明 | 架构基座与能力补齐期 |
| **v1.0** | 首个稳定版 | API 稳定承诺、配置兼容承诺、schema 向后兼容 | 生产可用里程碑 |
| **v2.0** | 重大演进 | 保留 v1 兼容期，新能力以可选模块叠加 | 智能化与生态扩展 |

### 1.2 发布节奏

- **v0.1 → v0.5**：基座先行，快速迭代。v0.1 完成架构分层与安全基线，v0.5 完成核心闭环与插件化基座。两个版本之间可有不兼容变更，但每个版本均附带迁移脚本。
- **v0.5 → v1.0**：能力补齐与稳定化。v1.0 是首个承诺 API/配置稳定的版本，发布前冻结破坏性变更。
- **v1.0 → v2.0**：稳定版之上的可选扩展。v2.0 新能力（AI/MCP/集群）以可选模块叠加，不破坏 v1.0 已承诺的接口。
- **每个版本必须可独立交付**：Docker 镜像可运行、核心闭环可用、不依赖下一版本才能使用。

### 1.3 风险控制原则

1. **基座先行**：架构分层、安全、并发、配置、迁移基座在 v0.1 完成，后续版本在此基础上叠加能力。
2. **向后兼容**：每个版本提供迁移脚本 + 兼容期，schema 变更走版本化迁移（`versioning.py`）。
3. **风险隔离**：高风险功能（如插件沙箱、AI Agent）独立版本验证，不与基座混杂。
4. **可验证交付**：每个版本定义明确交付标准，测试通过方可发布。

## 2. v0.1 版本规划

**对应**：架构基座与安全加固（P0 核心）
**主题**：完成架构分层、安全基线、并发加固、配置基座与迁移基座，为后续能力提供稳固地基。

### 2.1 新增功能

本版本交付 P0 中"基座与安全"类 6 项功能（Provider 抽象体系因复杂度高且为 P1 大量功能前置，作为 v0.5 的 P0 收尾）：

| 功能 | 编号 | 核心交付 |
|------|------|---------|
| 单文件巨石拆分（core.py 拆分） | P0-5 | 将 `core.py` 拆解为 `api/routes/services/domain/infra/providers` 五层，行为不变 |
| 配置分层与热重载基座 | P0-6 | 默认值→环境变量→配置文件→运行时覆盖四层叠加 + `post_save` 热重载 |
| 安全基线（bcrypt/CSRF/限流/脱敏/加密） | P0-2 | bcrypt 密码哈希、CSRF 防护、登录限流、敏感字段 Fernet 加密、API 响应脱敏、首启动强制改密 |
| SQLite 并发加固（短事务/锁重试/WAL） | P0-3 | WAL 模式 + `busy_timeout` + `retry_sqlite_locked` 指数退避 + 全链路短事务约束 |
| 后台任务运行时隔离 | P0-4 | APScheduler + asyncio 任务隔离，Provider 级 `throttle()` 限流 + 任务并发上限 |
| schema 版本化迁移与向后兼容 | P0-7 | 迁移脚本框架 + 现有表沿用 + 新增表（credentials/users/webauthn_credentials/task_sessions/plugins_state）迁移 |

### 2.2 重构内容

- **core.py 巨石拆分**（P0-5）：将单文件 `core.py` 按职责迁移到五层目录结构（`app/api/`、`app/routes/`、`app/services/`、`app/domain/`、`app/infra/`、`app/providers/`），新增 `domain/`（纯领域逻辑）与 `infra/`（基础设施）层。规避 nas-tools 单文件巨石被重构的反面（应避免 3.2）。
- **配置体系重构**（P0-6）：从 INI 单文件迁移至 `JsonConfigStore` 四层叠加 + mtime 缓存 + `post_save` 回调热重载。摒弃 aliyundrive-subscribe INI 单文件反面（应避免 3.10）。
- **安全贯穿改造**（P0-2）：Cookie/Token 写入路径统一接入 Fernet 加密；API 响应统一脱敏中间件；认证中间件覆盖所有 `/api/*`；首启动强制改密流程。
- **DB 层加固**（P0-3）：所有业务服务层短事务约束（不跨网络 IO 持锁）；Provider 限流锁与 DB 锁隔离。
- **后台运行时隔离**（P0-4）：Flask 同步阻塞改造为 FastAPI 异步 + APScheduler 后台隔离，规避 nas-tools 同步阻塞事件循环（应避免 3.1）。
- **迁移框架建立**（P0-7）：`app/versioning.py` 迁移脚本框架，迁移前自动备份 SQLite 文件，迁移事务化、失败可回滚。

### 2.3 兼容策略

- **配置迁移**：提供 `migrate_ini_to_json.py` 迁移脚本，将旧版 INI 配置转换为 `config/settings.json` 四层结构。迁移脚本幂等，可重复执行。
- **数据迁移**：现有表（resource_items/resource_jobs/subscription_*/monitor_*/local_files/scraper_*/recommendation_*/notification_dedupe/share_entries_cache）schema 沿用，仅新增表（credentials/users/webauthn_credentials/task_sessions/plugins_state）走版本化迁移。
- **破坏性变更标注**：
  - 🔴 **破坏性**：`core.py` 模块路径变更，外部直接 import 的代码需调整（提供 import 兼容期，v0.5 移除）。
  - 🔴 **破坏性**：Cookie/Token 存储格式由明文改为 Fernet 加密，首次启动自动迁移加密。
  - 🟡 **行为变更**：默认密码不再可用，首启动强制改密。
  - 🟡 **行为变更**：API 错误响应统一封装为 `{code,message,data}`，不再返回原始异常。
- **回滚保障**：迁移前自动备份 `data.db` → `data.db.bak.{timestamp}`，失败可回滚。

### 2.4 风险分析

| 风险点 | 等级 | 缓解措施 |
|--------|------|---------|
| 巨石拆分引入回归 bug | 高 | 先补集成测试覆盖现有行为、分阶段拆分、每阶段独立验证、保留向后兼容期 |
| Fernet 密钥管理不当（随镜像分发） | 高 | 密钥派生自 `JWT_SECRET` 环境变量、首启动强制改密、脱敏逻辑单测覆盖 |
| 部分 NAS 挂载文件系统不支持 WAL | 中 | 启动期检测并降级、可选 PostgreSQL（SQLAlchemy 抽象层）、`synchronous=NORMAL` |
| 迁移脚本本身有 bug 导致数据损坏 | 高 | 迁移前自动备份 SQLite 文件、迁移事务化、失败可回滚、保留 v1 兼容期 |
| 热重载误触发 Provider 重建中断进行中任务 | 中 | 热重载仅影响新请求、已建立 SSE/WebSocket 会话保持、敏感重建走平滑切换 |

### 2.5 交付目标

- [ ] 全部 6 项 P0 功能完成并验收通过。
- [ ] `core.py` 拆分完成，五层目录结构建立，行为不变（集成测试全绿）。
- [ ] 安全基线落地：bcrypt 密码、Fernet 加密、CSRF 防护、登录限流、首启动强制改密可用。
- [ ] SQLite WAL + 锁重试 + 短事务约束全链路落地，并发写测试通过。
- [ ] 后台任务与 Web 请求隔离，Provider 级限流生效，不阻塞事件循环。
- [ ] 配置四层叠加 + 热重载可用，`post_save` 回调驱动 Provider/调度/通知重载。
- [ ] schema 迁移框架可用，旧库可平滑迁移至新 schema，迁移失败可回滚。
- [ ] Docker 镜像可运行，`docker compose up` 一键启动可用。
- [ ] 单元测试覆盖率 ≥ 70%（domain 层 ≥ 90%），CI 流水线（pytest + ruff + mypy + Docker 构建）建立。

## 3. v0.5 版本规划

**对应**：插件化与体验补齐（P0 收尾 + P1 核心）
**主题**：完成 Provider 抽象体系（P0 收尾），落地插件化机制、能力声明、增量同步、Cookie 健康分层与任务会话日志等 P1 核心体验能力。

### 3.1 新增功能

本版本交付 P0 收尾 1 项 + P1 核心 6 项，共 7 项：

| 功能 | 编号 | 核心交付 |
|------|------|---------|
| Provider 抽象体系重构（含 DiscoveryProvider 接口） | P0-1 | 四类抽象基类（Cloud/Discovery/Metadata/Notify）+ 能力声明布尔开关 + 运行时注册表 + `config_fields` UI 渲染协议，迁移 5 网盘至新接口 |
| 插件化机制（五类插件） | P1-1 | Provider/Discovery/Metadata/Notify/Hook 五类插件统一接口 + 生命周期 + 进程内沙箱 + `PluginContext` 配置注入 + 目录扫描热加载 |
| 多网盘能力声明统一 | P1-2 | 能力布尔开关 + `config_fields` 字段定义 + 前端按定义动态渲染表单 + `GET /api/settings/providers` |
| 增量扫描 scan_token + mtime 比对 | P1-3 | scan_token 游标 + mtime 比对 + 仅拉变更集 + 增量 STRM 更新 + 能力声明 `supports_incremental_sync` |
| 流式解析降内存 | P1-4 | 分页拉取 + 流式解析 + 增量更新 + 断点续传，万级文件内存峰值可控 |
| Cookie 健康分层 + 加密存储 | P1-5 | `probe_connectivity` 探测 + 绿/黄/红健康分层 + SSE 推送 `cookie.health_changed` + 紧凑状态栏 UI |
| 按任务会话分页日志 | P1-6 | session_id 分配 + 独立日志文件 + 分页/流式 tail API + 结构化 JSON |

### 3.2 重构内容

- **Provider 抽象体系建立**（P0-1）：定义 `CloudProvider`/`DiscoveryProvider`/`MetadataProvider`/`NotifyProvider` 四类抽象基类，`supports_*` 能力声明布尔开关，`registry.py` 运行时注册表（对标 Alist `init()` 注册）。将现有 5 网盘（115/夸克/阿里/123/天翼）迁移至新接口。
- **插件体系建立**（P1-1）：`app/plugins/` 目录建立，`Plugin` 基类 + 生命周期（注册→加载→启用/禁用→卸载）+ `PluginManager` 沙箱隔离 + `PluginContext` 配置注入。与 `providers/`（内置）分离，`plugins/`（根目录）放第三方扩展。
- **能力声明驱动 UI**（P1-2）：前端通过 `GET /api/settings/providers` 获取 Provider 及 `config_fields`，动态渲染配置表单，新增 Provider 零侵入主流程（对标 Radarr UI 自动渲染）。
- **监控引擎增量改造**（P1-3）：`list_entries_incremental` 增量接口 + `local_files.scan_token` 游标 + mtime 比对 + 增量 STRM 更新。规避 Alist 全量 429 反面（应避免 3.12）。
- **流式处理改造**（P1-4）：监控/STRM/刮削引擎分页拉取 + 流式解析，`http_client.py` 流式接口。
- **Cookie 健康体系**（P1-5）：`credentials` 表 `health_status` 字段 + 按需探测 + 事件总线 `cookie.health_changed` + 通知联动 + 健康栏 UI。
- **日志系统重构**（P1-6）：`task_sessions` 表 + 按会话独立日志文件 `logs/task/{session_id}.log` + `GET /api/events/{session_id}/logs` 分页/流式 tail API + 结构化 JSON。

### 3.3 兼容策略

- **配置迁移**：插件配置独立命名空间（`plugins_state` 表），与现有 Provider 配置隔离，互不干扰。
- **数据迁移**：
  - `credentials` 表在 v0.1 已建立，v0.5 补充 `health_status` 探测逻辑与初始健康度填充。
  - `task_sessions` 表在 v0.1 已建立，v0.5 启用任务会话 ID 分配与日志写入。
  - `local_files` 表新增 `scan_token` 字段（若 v0.1 未含），迁移脚本填充空值，首次扫描时建立游标。
- **破坏性变更标注**：
  - 🔴 **破坏性**：`core.py` import 兼容期结束，移除兼容垫片。
  - 🟡 **行为变更**：网盘同步默认走增量（不支持增量的 Provider 降级全量 + 限流退避）。
  - 🟡 **行为变更**：Cookie 读取返回脱敏掩码，前端需适配健康状态展示。
- **Provider 接口迁移**：现有 5 网盘迁移至新 `CloudProvider` 接口，`supports_*` 默认值降低迁移成本，能力声明单测覆盖每个开关对应方法。

### 3.4 风险分析

| 风险点 | 等级 | 缓解措施 |
|--------|------|---------|
| Provider 抽象不当导致后续难以实现或能力泄漏 | 中 | 先以现有 5 盘验证接口完备性，参考 Alist `Storage` 与 Radarr Provider 字段收敛能力开关 |
| 第三方插件安全风险（恶意外联/资源耗尽/异常传播） | 高 | 默认禁用网络白名单外地址、`PLUGIN_TIMEOUT` 超时与内存限额、异常隔离降级、敏感操作二次确认 |
| scan_token 失效或部分网盘不支持增量 | 中 | 能力声明 `supports_incremental_sync`，不支持则降级全量 + 限流退避 |
| 能力声明与实际实现不一致（声明支持但方法未覆写） | 中 | 能力声明单测 + Provider mock 测试覆盖每个 `supports_*` 对应方法 |
| 流式中断导致数据不一致 | 中 | 分页幂等 + 游标续传 + 失败重试 |
| 探测本身触发网盘风控 | 中 | 按需探测（用户触发/任务前探测）+ 限流 + 失败退避，避免高频轮询 |
| 日志文件过多占用磁盘 | 低 | 保留 N 天自动清理 + 引擎汇总日志便于跨任务运维概览 |

### 3.5 交付目标

- [ ] P0-1 Provider 抽象体系完成，5 网盘迁移至新接口，能力声明单测全绿。
- [ ] P1 插件化机制（6 项）完成，五类插件接口可用，沙箱隔离生效，热加载可用。
- [ ] 能力声明驱动 UI 可用，新增 Provider 无需改前端代码。
- [ ] 增量扫描落地，万级文件扫描耗时与请求量显著下降（对标 qmediasync 增量缓存）。
- [ ] Cookie 健康分层可视化，绿/黄/红状态栏可用，健康变更 SSE 推送生效。
- [ ] 按任务会话分页日志可用，`GET /api/events/{session_id}/logs` 流式 tail 生效。
- [ ] 五位一体闭环（多网盘 + STRM + TG 资源同步 + 影视订阅 + 刮削管理）可在新架构上端到端跑通。
- [ ] Docker 镜像可运行，插件目录扫描加载可用。

## 4. v1.0 版本规划

**对应**：稳定版与差异化（P1 收尾 + P2 核心）
**主题**：完成 P1 剩余差异化能力（豆瓣防封禁/可观测性/转存任务组/302 代理），落地 P2 核心稳定版能力（多用户/插件市场/Passkey/主题/移动端/Override Rules），冻结 API 与配置兼容承诺。

### 4.1 新增功能

本版本交付 P1 收尾 4 项 + P2 核心 6 项，共 10 项：

| 功能 | 编号 | 核心交付 |
|------|------|---------|
| 豆瓣元数据 + 防封禁框架 | P1-7 | 节流 + 代理 + 重试 + 缓存 + 双源融合（豆瓣中文 + TMDB 季集）+ 反爬持续应对 |
| 可观测性 Server-Timing | P1-8 | Server-Timing 头暴露内部耗时（db/provider/total）+ request_id 贯穿 + 结构化日志 |
| 转存任务组 + 失效检查 + 新链挑选 | P1-9 | 任务组调度 + 失效探测 + 新链挑选 + 集数账本联动 + 评分算法 |
| 302 直链代理 + UA 适配 | P1-10 | 内置轻量 HTTP 代理端口 + UA/鉴权头改写 + 直链优先/代理兜底 |
| 多用户与 RBAC | P2-1 | admin/user/guest 三角色 + 权限位 + 多用户配额 + RBAC 扩展 |
| 插件市场/插件管理 UI | P2-2 | 市场清单 + 一键安装 + 启用/禁用/配置 UI + 插件评分 |
| WebAuthn Passkey 登录 | P2-3 | WebAuthn 注册/认证流程 + 多设备管理 + 密码/Passkey 并存兜底 |
| 主题/夜间模式完善 | P2-4 | Tailwind dark mode + 主题切换 + 偏好持久化 + WCAG 对比度 |
| 移动端适配增强 | P2-5 | 响应式重构 + 触控优化 + 移动端导航 + 核心功能移动可用 |
| Override Rules 条件化请求默认值 | P2-7 | 规则引擎 + 用户/标签条件化默认值 + 优先级 + 冲突检测（依赖 P2-1） |

### 4.2 重构内容

- **刮削引擎双源改造**（P1-7）：`MetadataProvider` 抽象落地 TMDB + 豆瓣双源，`douban.py` 防封禁框架（节流/代理/重试/缓存），双源互补降级。规避 MoviePilot/metashark 强依赖豆瓣无防封禁反面（应避免 3.9）。
- **可观测性中间件**（P1-8）：`ServerTimingMiddleware` + 计时埋点 + `request_id` 贯穿日志与 Server-Timing，`metrics.py` 指标暴露（预留 Prometheus）。
- **订阅引擎增强**（P1-9）：`ShareSelection` 新链挑选 + 评分算法 + 集数账本最优命中校验 + 内容指纹去重。借鉴 quark-auto-save 转存任务组。
- **STRM 引擎 302 代理**（P1-10）：内置轻量 HTTP 代理端口 + UA/鉴权头改写 + 直链优先代理兜底。借鉴 qmediasync 内置 115 下载链接代理。
- **认证体系扩展**（P2-1/P2-3）：`users` 表多用户 + RBAC 权限位 + `webauthn_credentials` 表 Passkey 注册/认证流程。
- **插件市场建立**（P2-2）：`marketplace.py` 插件市场清单 + 一键安装 + 管理 UI（对标 MoviePilot 300+ 插件）。
- **表现层完善**（P2-4/P2-5）：Tailwind 暗色模式 + 移动端响应式重构。
- **规则引擎建立**（P2-7）：Override Rules 条件化默认值（对标 Seerr）。

### 4.3 兼容策略

- **API 稳定承诺**（v1.0 核心承诺）：

### 7.2 关键依赖链路说明

1. **基座先行链路**（v0.1）：P0-5（巨石拆分）是绝大多数功能的隐性前置——分层不清则后续模块无处安放。P0-6（配置热重载）与 P0-2（安全基线）依赖 P0-5 落地后的分层结构。P0-3（SQLite 加固）→ P0-4（后台隔离）→ P0-7（schema 迁移）构成数据与运行时基座链。

2. **核心闭环链路**（v0.5）：P0-1（Provider 抽象）是核心闭环的架构前提，P1 大量功能（插件化/能力声明/增量/健康分层/转存组/302 代理）都依赖它。P0-1 作为 v0.5 的 P0 收尾，为 P1 核心能力提供承载。

3. **插件链路**（v0.5 → v1.0）：P1-1（插件化机制）→ P2-2（插件市场 UI），前者定义接口与沙箱（v0.5），后者提供分发与管理（v1.0）。

4. **安全贯穿链路**（v0.1 → v1.0）：P0-2（安全基线，v0.1）是 P1-5（Cookie 健康分层，依赖 credentials 加密表，v0.5）与 P2-1/P2-3（RBAC/Passkey，v1.0）的前置。

5. **AI 链路**（v2.0）：P3-2（MCP）+ P3-3（事件驱动完善）共同支撑 P3-1（AI Agent），任一缺失则 Agent 无法编排面板能力。P3-2 依赖 P1-2 能力声明（将能力暴露为 MCP 工具）。

6. **分布式链路**（v2.0）：P3-6（集群）依赖 P0-3（SQLite→PG 切换）与 P3-3（分布式事件总线），是最终演进终点。

7. **稳定承诺边界**：v1.0 是稳定承诺的分水岭——v0.x 可有不兼容变更（附迁移脚本），v1.0 起 API/配置冻结，v2.0 新能力以可选模块叠加不破坏 v1.0 已承诺接口。

- **v0.5 → v1.0**：能力补齐与稳定化，v1.0 前冻结破坏性变更。
- **v1.0 → v2.0**：稳定版之上的可选扩展，不破坏 v1.0 已承诺接口。
